Patterns 1–4: Why Each Was Chosen

01 Facade
Facade Pattern
KioskInterface → purchaseItem() · refundTransaction() · runDiagnostics() · restockInventory()
External systems — city dashboards, mobile apps, operator consoles — must never reach into KioskCore, the command system, the inventory manager, or the hardware layer directly. KioskInterface is the one and only entry point. It delegates internally to the right subsystem but never exposes how. Adding a new internal subsystem never breaks the external API contract.
Constraint Resolved "External systems should interact only with this interface." The Facade directly enforces that boundary — nothing outside the diagram calls into KioskCore or lower layers directly.
core/kiosk_interface.py core/kiosk_core.py
02 Singleton
Singleton Pattern
CentralRegistry · EventBus → getInstance() always returns the same object
System configuration, kiosk registration, and the event bus must be consistent across every part of the system. Creating multiple instances would produce conflicting state — two registries with different kiosk maps, or two event buses with different subscribers. CentralRegistry uses Python's __new__ with a double-checked threading.Lock to guarantee exactly one instance, even under concurrent construction attempts.
Constraint Resolved Spec: "Implement a CentralRegistry that stores global system information such as configuration and system status." The Singleton ensures this is uniquely accessible without being manually passed between every class.
core/central_registry.py
03 Abstract Factory
Abstract Factory Pattern
KioskFactory → PharmacyKioskFactory · FoodKioskFactory · EmergencyKioskFactory
A pharmacy kiosk needs a prescription verifier, a sealed dispenser, and a controlled inventory policy. An emergency kiosk needs an open verifier, a basic dispenser, and a rationed inventory policy with a hard cap. Mixing components from different families would produce a broken kiosk. The Abstract Factory guarantees components are always assembled as a compatible set — KioskCore never knows which concrete types it received.
Constraint Resolved "Different kiosk types must be created — PharmacyKiosk, FoodKiosk, EmergencyReliefKiosk — each factory creates compatible components." Abstract Factory is the only pattern that produces families of related objects in one coordinated step.
factory/kiosk_factory.py factory/pharmacy_kiosk_factory.py factory/food_kiosk_factory.py factory/emergency_kiosk_factory.py
04 State
State Pattern
KioskMode → ActiveMode · PowerSavingMode · MaintenanceMode · EmergencyLockdownMode
Without the State pattern, KioskCore.handleRequest() would contain a giant if/switch checking the current mode before every action. The State pattern pushes that branching into each mode class. Adding a new mode requires zero changes to KioskCore. Each mode completely encapsulates its own behaviour for purchases, refunds, restock, and diagnostics. EmergencyLockdownMode overrides all others immediately when the emergency event fires.
Constraint Resolved "The kiosk must support different operating states — Active, Power-saving, Maintenance, Emergency lockdown. Each state changes how the kiosk behaves when users interact with it." Also satisfies the event priority constraint: EmergencyLockdownMode takes over instantly.
modes/kiosk_mode.py modes/active_mode.py modes/power_saving_mode.py modes/maintenance_mode.py modes/emergency_lockdown_mode.py
